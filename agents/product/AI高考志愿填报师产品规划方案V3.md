## 一、产品愿景与定位

### 1.1 产品名称与Slogan

| 项目 | 内容 |
|------|------|
| 产品名称 | 志愿超级副驾 |
| 英文名 | Zhiyuan Copilot |
| Slogan | **"懂志愿，更懂你"** |
| 辅助Slogan | "AI帮你搞懂志愿填报" |

**命名逻辑：**
- "超级副驾"传递伙伴定位（非替代、而是增强）、科技感与可信赖感
- 区别于市面"AI替你选"的工具型产品，强调理解力与陪伴感

---

### 1.2 核心定位

> **面向高考志愿填报师的B端SaaS工具，帮助咨询师提升效率、生成专业报告**

**战略定位：从"AI替你选"转向"AI帮你搞懂志愿填报"**

| 维度 | 旧范式（竞品） | 志愿超级副驾 |
|------|-------------|------------|
| 核心逻辑 | AI直接输出志愿结果 | AI辅助咨询师决策 |
| 用户关系 | AI→学生（替代） | 咨询师→学生（AI赋能） |
| 价值主张 | 志愿结果 | 理解能力+效率工具 |
| 风险边界 | 结果错误责任模糊 | 咨询师始终为决策主体 |
| 监管态度 | 高风险（算法黑盒） | 低风险（辅助工具定位） |

**一句话定位：**
> 高考志愿填报师的智能工作台——不是替代你的判断，而是放大你的能力。

---

### 1.3 目标用户画像

#### 核心用户：高考志愿填报师

**画像A：独立咨询师**
- 年龄：25~40岁
- 背景：教育学/心理学背景，或跨界转型（教培从业者、家有考生的家长）
- 规模：1~3人，无/轻团队
- 年服务学生数：50~300人/年
- 核心场景：高考出分后1~2周的高峰咨询期
- 痛点：时间极度紧张，需要快速生成个性化方案和专业报告
- 付费意愿：中等（年费2000~5000元区间可接受）
- 工具习惯：微信+Excel+纸质资料，数字化程度低

**画像B：中小型机构/工作室**
- 规模：3~15人咨询师团队
- 背景：区域性教育机构、高考复读学校招生部门
- 年服务学生数：300~2000人/年
- 核心场景：批量处理志愿方案，需要团队协作和报告导出
- 痛点：多人协作效率低、报告标准不统一、数据管理混乱
- 付费意愿：较高（年费5000~20000元区间可接受）
- 工具习惯：有基础的教务管理系统，期望与现有工作流集成

#### 用户决策链

```
决策者（机构负责人）
    ↓ 采购决策（价格、品牌背书）
使用者（咨询师）
    ↓ 日常使用（体验、功能、数据）
最终受益人（学生/家长）
    ↓ 感知价值（报告质量、沟通效率）
```

### 2.2 目标用户痛点分析

#### 核心痛点（HP1~HP3）

**HP1：时间严重不对称 ⭐⭐⭐⭐⭐**
> "出分到截止只有7~10天，我一个人根本服务不过来"

- 高考出分后咨询量激增，单个咨询师单日接待上限约15~20组家庭
- 每组家庭的个性化方案制作耗时2~4小时
- 峰值期间日工作时长超过16小时，极度疲劳导致决策质量下降

**HP2：信息差与数据陈旧 ⭐⭐⭐⭐**
> "各高校的招生章程、专业分数线、排位变化，我根本追不过来"

- 全国3000+高校，每年招生章程均有调整
- 历年录取数据分散在不同渠道，更新滞后
- AI加工的数据质量参差不齐，错误数据误导决策风险高

**HP3：报告专业性不足 ⭐⭐⭐⭐**
> "家长要的是看得懂的报告，不是密密麻麻的Excel"

- 手写/Excel方案专业感不足，家长信任度低
- 缺乏可视化呈现（录取概率图、梯度分析图）
- 机构无法统一输出标准化的服务报告

#### 次要痛点（HP4~HP6）

| 编号 | 痛点 | 严重程度 | 说明 |
|------|------|---------|------|
| HP4 | 团队协作效率低 | ⭐⭐⭐ | 多人处理同一学生，数据不互通 |
| HP5 | 志愿填报规则理解门槛高 | ⭐⭐⭐ | 新人咨询师培养成本高 |
| HP6 | 家长沟通成本高 | ⭐⭐ | 反复解释同一概念，消耗精力 |

### 2.3 竞争格局分析

#### 竞品分类

| 类型 | 代表产品 | 定位 | 优势 | 劣势 |
|------|---------|------|------|------|
| **C端工具型** | 夸克志愿、知乎知料、掌上高考 | 直接服务学生/家长 | 流量大、品牌强 | 定位与我方不冲突，但蚕食C端市场 |
| **B端工具型** | 志愿填报系统（高校处购买） | 高校招生办使用 | 数据全 | 非商业化、体验差、不面向咨询师 |
| **咨询机构自建** | 好未来、新东方内部系统 | 自用 | 定制化 | 不对外、研发成本高 |
| **新兴AI工具** | 各创业团队产品 | 面向C端 | 技术新 | 合规风险高、数据质量差 |

#### 竞争差异化矩阵

| 维度 | 志愿超级副驾 | C端工具 | 传统B端系统 |
|------|------------|--------|-----------|
| 目标用户 | B端咨询师 | C端学生 | 高校/招办 |
| 核心价值 | 效率+专业报告 | 快速出结果 | 数据管理 |
| AI介入程度 | 辅助（咨询师主导） | 替代决策 | 无AI |
| 报告输出 | 专业级可交付报告 | 仅供参考 | 无 |
| 合规设计 | 从一开始就合规 | 高风险 | 不适用 |
| 价格策略 | SaaS订阅制 | 免费/低价 | 一次性采购 |

#### 竞争壁垒分析

1. **数据壁垒**：高考录取数据的完整性和鲜度需要长期积累
2. **场景壁垒**：深度理解B端咨询师工作流程，非通用AI能力可替代
3. **信任壁垒**：B端用户切换成本高，服务稳定性和数据安全是长期护城河
4. **合规壁垒**（差异化优势）：率先完成算法备案，建立监管护城河

### 3.2 核心功能详细描述

#### 3.2.1 智能院校推荐引擎

**输入：** 考生省份、科类、分数、位次、选科、意向专业/院校列表（可多选）

**输出：** 三梯度院校推荐列表（冲/稳/保）+ 推荐理由

**推荐逻辑（五层架构）：**
1. **基础层**：基于考生分数位次，匹配近3年录取数据
2. **规则层**：应用各省志愿填报规则（平行志愿/顺序志愿/专业级差等）
3. **偏好层**：结合意向专业、城市、学校层次进行加权排序
4. **风险层**：计算各院校录取概率，过滤风险过高/过低选项
5. **解释层**：为每个推荐结果生成AI解读（"为什么推荐/为什么不推荐"）

**推荐结果展示：**
- 每所院校显示：院校名称、预估录取概率（区间）、近3年录取位次、专业组信息
- 每所院校附带推荐理由（AI生成，200字以内）
- 支持手动调整梯度、排除院校、锁定院校

---

#### 3.2.2 专业级报告生成

**报告结构（六章）：**

```
封面
├── 第一章：考生基本情况
│   ├── 高考成绩与位次
│   ├── 选科组合分析
│   └── 学业优势与特质
├── 第二章：志愿方案设计
│   ├── 冲/稳/保院校列表（含推荐理由）
│   ├── 专业组选择分析
│   └── 梯度合理性评估
├── 第三章：风险提示
│   ├── 滑档风险评估
│   ├── 专业调剂风险提示
│   └── 政策变动注意事项
├── 第四章：备选方案
│   └── 方案B/C 简述
├── 第五章：数据说明
│   └── 数据来源、更新时间、局限性说明
└── 附录：合规声明、免责声明
```

**报告特点：**
- 支持机构品牌定制（Logo、联系方式、水印）
- 支持一键生成H5链接（微信分享）
- 支持导出PDF（高清/标准双版本）
- 报告带唯一编号，防伪可查

#### 3.2.3 录取概率模拟推演

**核心场景：** 当考生最心仪的院校（冲）今年分数上涨时，系统快速模拟调整方案的影响。

- 输入假设："A院校录取分数+10分，B专业组录取概率变化"
- 输出：新的梯度调整建议列表，并标注影响程度（高/中/低）
- 可对比多个假设场景，生成对比分析图

### 6.2 数据合规框架

#### 法律依据

| 法规 | 核心要求 | 我们的对应措施 |
|------|---------|-------------|
| 《个人信息保护法》 | 个人信息收集需告知+同意 | 用户协议明确告知，敏感信息（分数）需单独确认 |
| 《数据安全法》 | 数据分类分级保护 | 将数据分为公开数据/用户数据/加工数据三类，分类存储和访问控制 |
| 《生成式AI服务管理暂行办法》 | 内容合规、算法备案 | 报告内容合规审核机制，算法按期完成备案 |
| 《互联网信息服务算法推荐管理规定》 | 算法透明、可关闭 | 推荐逻辑可解释，提供非个性化选项 |

#### 合规红线（绝对禁止）

1. **不收集18岁以下未成年人信息** — B端产品，用户是咨询师，不直接接触未成年人
2. **不存储原始考生分数用于模型训练** — 分数数据仅用于实时计算，算完即删
3. **不使用志愿推荐结果做商业变现** — 用户生成的报告和数据归属用户/机构
4. **不发布误导性录取承诺** — 所有概率表述均为区间值，禁止"百分百录取"类表述

### 6.3 数据质量保障

#### 质量标准

| 指标 | 标准 | 检测方式 |
|------|------|---------|
| 数据准确率 | ≥99.5%（历年录取分数线） | 每年9月与官方数据人工抽查比对 |
| 数据完整率 | ≥98%（目标院校覆盖率） | 系统自动覆盖率统计 |
| 数据鲜度 | 招生季前30天完成最新数据更新 | 版本管理+发布时间戳 |
| 响应时间 | 数据查询≤2秒，报告生成≤30秒 | 服务端SLA监控 |

#### 质量保障机制

1. **双源交叉验证**：同一数据点至少从2个独立来源确认
2. **人工审核节点**：每年4~5月（招生章程发布期）启动全员数据审核
3. **用户纠错通道**：用户发现数据错误可一键提交，核实后奖励积分
4. **版本快照**：每个招生年度数据独立版本存档，支持回溯

### 7.2 算法推荐管理规定合规

#### 合规要求对照

| 规定条款 | 要求 | 我们的情况 |
|---------|------|---------|
| 算法备案 | 具有舆论属性或社会动员能力的算法需备案 | 明确用于B端辅助决策，非舆论属性，申请备案 |
| 算法透明 | 告知用户算法基本原理 | 隐私政策中披露推荐逻辑，用户可查看推荐理由 |
| 算法可关闭 | 提供不基于算法的选项 | 专业版及以上提供"手动筛选模式" |
| 算法审计 | 定期自我评估，保存记录 | 每半年做算法自审，记录存档 |
| 拒绝违规内容 | 不得生成违规推荐结果 | 推荐理由AI生成内容经过内容安全审核层 |

#### 算法备案评估

```
备案类型：工具类辅助算法（非舆论/社交类）
备案难度：中等（需明确辅助定位，区别于替代决策类应用）
预计周期：2~3个月
备案成本：材料准备+审核配合，约2人/月工作量
风险评估：低（定位清晰，无监管禁止场景）
```

### 7.3 隐私保护方案

#### 隐私保护设计原则

| 原则 | 具体措施 |
|------|---------|
| **最小化** | 仅收集功能必需的数据，不收集与志愿填报无关的信息 |
| **本地处理** | 考生分数等敏感数据在前端处理，不上传服务器（除非用户主动存储） |
| **权限隔离** | 机构内不同咨询师的数据严格隔离，机构管理员可见团队汇总，不可见个体详情 |
| **删除权保障** | 用户可随时删除其录入的案例数据，7天内完成物理删除 |
| **传输加密** | 全站HTTPS，敏感数据传输使用TLS1.3 |
| **存储加密** | 用户敏感数据（姓名、分数）AES-256加密存储 |

#### 隐私政策关键条款

1. 数据控制权归用户/机构所有
2. 不将用户数据用于模型训练（明确承诺）
3. 不与第三方共享可识别个人的数据
4. 第三方SDK使用最小化，仅使用必要的统计分析工具
5. 18岁以下儿童信息：严格限制，不主动收集

### 8.2 应对策略

#### R1：政策监管风险

**应对策略：**
1. 从产品定位上明确"辅助工具"而非"替代决策"定位
2. 所有宣传材料不使用"保证录取"等违规表述
3. 主动与行业协会沟通，争取成为合规示范案例
4. 算法备案提前准备，上线后3个月内完成

**触发条件：** 监管文件明确针对AI志愿推荐应用 → 立即停止相关营销表述，产品增加人工确认流程

---

#### R2：数据质量风险

**应对策略：**
1. 建立数据双人审核机制，错误数据纳入KPI考核
2. 高考出分前（5月）冻结当年数据，禁止随意修改
3. 报告底部注明"数据仅供参考，以教育考试院官方数据为准"
4. 用户发现错误一键纠错，核实后48小时内更新

---

#### R3：数据安全风险

**应对策略：**
1. 通过等保二级或ISO 27001认证（Y1完成）
2. 数据存储与计算分离，最小化敏感数据在服务器的留存时间
3. 每半年委托第三方进行渗透测试
4. 购买网络安全保险（保额不低于500万元）

---

#### R4：市场接受度风险

**应对策略：**
1. 早期通过免费/低价获取种子用户，用真实数据证明ROI
2. 提供按月订阅而非强制年付，降低用户决策门槛
3. 强调"节省的时间价值"而非"工具本身"，对标咨询师时薪
4. 建立成功案例库，用真实咨询师使用前后对比说话

---

#### R5：竞争加剧风险

**应对策略：**
1. 深耕B端场景，做深度而非广度（大厂难以为单一行业做深度定制）
2. 积累咨询师网络，构建人脉护城河（社区+口碑）
3. 快速完成合规备案，形成合规先发优势
4. 持续迭代数据质量和功能深度，保持技术代差

---

#### R6：技术依赖风险

**应对策略：**
1. 同时接入2~3家大模型API（OpenAI/Claude/国内GLM），随时可切换
2. 核心推荐算法自研，不依赖单一API能力
3. 探索模型蒸馏，将大模型能力迁移到小模型，降低调用成本
4. 设定API成本上限预警机制

---

#### R7：季节性波动风险

**应对策略：**
1. 拓展非高考咨询场景：考研志愿、留学申请、中考志愿（地方性）
2. 非高峰季重点推进：社区运营、内容积累、功能迭代
3. 设计"高考季准备期套餐"，让用户在淡季提前续费锁定权益
4. 发展知识付费（课程、培训），形成非季节性收入

---

#### R8：人才流失风险

**应对策略：**
1. 核心算法和数据工程师签署竞业协议（按法规给予补偿）
2. 关键数据文档化、代码结构化，降低单点依赖
3. 核心团队成员给予股权激励，长期绑定
4. 建立AB角制度，确保关键岗位有备份
