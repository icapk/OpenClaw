## 一、系统架构

### 1.1 整体设计：单进程网关模式

OpenClaw 采用了**单一长期运行进程（Gateway）** 作为核心，所有消息通道、Agent 运行时、控制界面都连接到这个网关。

```
┌─────────────────────────────────────────────────────────┐
│                    OpenClaw Gateway                      │
│                                                         │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌──────────┐   │
│  │WhatsApp │  │Telegram │  │Discord  │  │ iMessage  │   │
│  │Channel  │  │Channel  │  │Channel  │  │ Channel   │   │
│  └────┬────┘  └────┬────┘  └────┬────┘  └────┬─────┘   │
│       │            │            │            │         │
│  ┌────┴────────────┴────────────┴────────────┴────┐    │
│  │              Channel Registry                   │    │
│  └────────────────────┬───────────────────────────┘    │
│                       │                                 │
│  ┌────────────────────┴───────────────────────────┐    │
│  │           Plugin Registry (Capability Map)       │    │
│  └────────────────────┬───────────────────────────┘    │
│                       │                                 │
│  ┌────────────────────┴───────────────────────────┐    │
│  │              Pi Agent Runtime                   │    │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────────────┐ │    │
│  │  │ Workspace│  │ Sessions│  │  Memory/Skills  │ │    │
│  │  │  Files   │  │  Store  │  │   Management    │ │    │
│  │  └─────────┘  └─────────┘  └─────────────────┘ │    │
│  └───────────────────────────────────────────────┘    │
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │  WebSocket Server (port 18789)                   │   │
│  │  - macOS/iOS/Android Nodes                       │   │
│  │  - CLI / Control UI / macOS App                 │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

**为什么这样设计？**

这是最适合"个人 AI 助手"场景的架构。单进程意味着：
- 零配置开箱即用（不需要安装 Redis/ZooKeeper 等外部依赖）
- 所有状态在一个进程内，调试简单
- 资源占用低，一台 MacBook 足以运行

但这也是它最大的扩展性瓶颈（见短板分析）。

### 1.2 核心模块关系

| 模块 | 职责 | 关键文件模式 |
|------|------|-------------|
| **Gateway** | 入口，维护 WebSocket 服务，持有所有 channel 连接 | `dist/index.js`, `dist/entry.js` |
| **Plugin Registry** | 中心化注册表，所有插件通过 `register()` 接入 | `dist/plugin-sdk/` |
| **Agent Runtime** | Pi agent 核心封装，工具调用、会话管理 | `@mariozechner/pi-agent-core` |
| **Channel Plugins** | 各平台协议接入（WhatsApp/Telegram/Discord） | `dist/extensions/<channel>/` |
| **Provider Plugins** | 模型提供商接入（OpenAI/Anthropic/Google） | `dist/extensions/<provider>/` |
| **Node Protocol** | iOS/Android/macOS 节点接入 | `dist/widearea-dns-*.js` |
| **Control UI** | 浏览器控制台 | `dist/control-ui/` |

### 1.3 插件系统设计

OpenClaw 的插件系统是其最具工程价值的部分，采用**声明式能力注册**模式：

**三层架构：**
1. **Manifest（清单层）** — `openclaw.plugin.json` 定义插件元数据，无需执行代码即可读取
2. **Registry（注册层）** — 插件调用 `api.registerXxx()` 向中心注册表声明能力
3. **Runtime（运行时层）** — 核心从 Registry 读取能力并消费

**能力类型（Capability）：**
```
Text Inference        → registerProvider()
Speech Synthesis      → registerSpeechProvider()
Media Understanding    → registerMediaUnderstandingProvider()
Image Generation      → registerImageGenerationProvider()
Web Search            → registerWebSearchProvider()
Channel/Messaging     → registerChannel()
```

**设计哲学："插件拥有厂商实现，核心拥有能力契约"**

这不是一句空话。它的实际含义是：
- OpenAI 插件拥有 OpenAI 的全部接入（文本、语音、图像、搜索）
- 但**通用逻辑**（如何选择 TTS 提供商、如何做 failover）在核心
- 频道插件（如 Discord）不知道也不需要知道背后用哪个模型

这使得添加新厂商或换掉现有厂商时，核心代码**不需要改一行**。

### 1.4 Agent 管理与通信

**会话模型：**
- 路径：`~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`
- 格式：JSONL（每行一个消息对象）
- Key 格式：`agent:<agentId>:<mainKey>`（如 `agent:main:feishu:direct:ou_xxx`）

**多 Agent 路由（Bindings）：**
```json
bindings: [
  { agentId: "alex", match: { channel: "whatsapp", peer: { kind: "direct", id: "+15551230001" } } },
  { agentId: "mia",  match: { channel: "whatsapp", peer: { kind: "direct", id: "+15551230002" } } }
]
```
- 路由规则：最具体匹配优先（peer > guildId > accountId > channel > default）
- 每个 Agent 有独立 Workspace、Auth Profiles、Session Store

**工具调用流程：**
```
Inbound Message → Binding → Agent Session → Pi Agent Core → Tools
                                                        ↓
                                              Plugin Registry 读取
                                                        ↓
                                              Channel 执行（发送消息）
```

### 1.5 配置系统

**配置文件：** `~/.openclaw/openclaw.json`（JSON5 格式）

**配置层级：**
1. 环境变量覆盖（`OPENCLAW_*`）
2. `openclaw.json` 用户配置
3. 默认值（内嵌）

**Schema 验证：** 使用 TypeBox + AJV，配置在**执行前**验证，早期发现错误。

## 二、技术栈

### 2.1 核心依赖

| 库 | 用途 |
|----|------|
| **Node.js 22+** | 运行时 |
| **Hono** | HTTP/WebSocket 网关服务器（非 Express！） |
| **@modelcontextprotocol/sdk** | MCP 协议支持 |
| **@mariozechner/pi-agent-core** | Agent 核心逻辑 |
| **@sinclair/typebox** | 运行时类型定义 + JSON Schema 生成 |
| **jiti** | Node.js 中直接执行 TypeScript（无需编译） |
| **chokidar** | 文件监控（workspace 文件热重载） |
| **croner** | 定时任务（Cron Jobs） |
| **ajv** | JSON Schema 验证 |
| **@line/bot-sdk** | LINE 消息 |
| **@lydell/node-pty** | 终端模拟（exec 工具） |
| **chalk** | 命令行彩色输出 |
| **commander** | CLI 参数解析 |
| **json5** | JSON5 解析（配置文件） |
| **dotenv** | 环境变量 |

### 2.2 为什么选 Hono 而不是 Express？

这是值得注意的技术决策。Hono 是轻量级（~14KB）、基于 Web Standards 的框架，比 Express 快 3-5 倍。OpenClaw 选择 Hono 意味着：
- WebSocket 处理更高效
- 中间件链更轻
- 但生态插件没有 Express 丰富（不过 OpenClaw 自己不依赖很多中间件）

### 2.3 Pi Agent Core 的角色

`@mariozechner/pi-agent-core` 是 OpenClaw 的"大脑"。它负责：
- LLM 调用和流式响应
- 工具执行引擎
- Prompt 构建和上下文管理
- 会话状态维护

OpenClaw 在它之上构建了：多通道路由、插件集成、文件 workspace、记忆系统。

## 三、代码结构

### 3.1 目录组织

```
openclaw/dist/
├── index.js              # 主入口（CLI）
├── entry.js              # Gateway 进程入口（带 respawn 策略）
├── run-main-*.js         # 主命令路由
├── library-*.js          # 共享库（配置/会话/端口管理）
│
├── agents/               # Agent 相关（auth/session/runtime）
│
├── extensions/           # 插件扩展（bundled）
│   ├── anthropic/        # Anthropic 模型提供商
│   ├── openai/           # OpenAI 模型提供商
│   ├── discord/          # Discord 频道
│   ├── telegram/         # Telegram 频道
│   ├── whatsapp/         # WhatsApp 频道
│   ├── acpx/            # ACP 协议（OpenClaw 内部代理协议）
│   └── ...
│
├── plugin-sdk/           # 插件 SDK（对外 API）
│   ├── plugin-entry.js  # 插件入口定义
│   ├── core.js          # 核心 API（registerXxx）
│   ├── channel-*.js     # 频道相关 SDK
│   ├── provider-*.js    # 提供商相关 SDK
│   └── ...
│
├── bundled/              # 内置 Bundled 插件
│   ├── session-memory/
│   ├── command-logger/
│   └── ...
│
├── control-ui/          # Web 控制台（静态资源）
│   └── index.html
│
└── infra/                # 基础设施（warning-filter）
```

### 3.2 核心入口文件

**CLI 入口**：`dist/index.js` → `dist/entry.js` → `dist/run-main-WhPPYnun.js`

流程：
1. `entry.js` 处理 respawn（Node.js 选项、TLS 证书）
2. 解析 CLI 参数（profile/container 模式）
3. 加载 Legacy CLI deps
4. 调用 `runCli()`

**Gateway 入口**：`openclaw gateway` 命令
- 启动 Hono WebSocket 服务器（默认 `127.0.0.1:18789`）
- 初始化 Plugin Registry
- 启动所有已配置 Channel

### 3.3 重要设计模式

**① Manifest-First 插件发现**
```
Startup → 扫描插件目录 → 读取 manifest（不执行代码）→ 验证配置 → 决定启用/禁用 → 加载运行时
```
好处：配置错误在真正运行前就能发现，不会在启动到一半时崩溃。

**② Capability Registry 模式**
```
Plugin.registerXxx() → Central Registry ← Core Runtime 读取
```
单向数据流，插件不能直接调用核心内部状态，只能通过 registry 交互。

**③ 协议类型生成（TypeBox + Codegen）**
```
TypeBox Schema → JSON Schema → Swift Models
```
协议是类型安全的，Swift 客户端和 Gateway 共享同一套 schema。

**④ 设备配对 + Token 信任链**
```
Device Connect → Challenge/Response → Pairing Approval → Device Token
```
非本地连接需要设备配对 Approval，防止未授权设备接入。

## 四、产品对比分析

### 4.1 核心优势

#### ✅ 1. 真正的"个人 AI 助手"定位
没有试图做"企业协作平台"，而是专注于让个人在所有日常聊天应用中拥有一个 AI 助手。这是最难也是最有价值的场景之一。

#### ✅ 2. 零外部依赖的单进程部署
不需要 Redis、PostgreSQL、Kafka——开箱即用。这对于个人用户和技术爱好者来说是巨大的摩擦降低。

#### ✅ 3. 多通道统一接入
一个 Gateway 管理所有聊天应用，路由规则集中配置。相比分别搭建 Telegram Bot + Discord Bot + WhatsApp Web，要省心得多。

#### ✅ 4. 插件能力系统设计优秀
"核心拥有能力契约，插件拥有厂商实现"的哲学非常正确。这使得 OpenClaw 可以轻松接入任何新模型提供商或聊天平台，而不需要改动核心代码。

#### ✅ 5. 多 Agent 隔离
Binding 路由系统允许一台设备服务多个人，每人拥有独立的 Workspace、Session、Auth。这是很多竞品做不到的精细度。

#### ✅ 6. 移动节点支持
iOS/Android 节点可以提供 Canvas、Camera、Screen Recording 能力，这是差异化亮点。想象一下"给 AI 发一张照片问这是什么"——这个场景很多竞品做不了。

#### ✅ 7. 自托管隐私
数据完全在用户自己机器上，没有云端中转。对于有隐私意识的技术用户，这是决定性因素。

#### ✅ 8. Workspace 文件即记忆
用 Markdown 文件做记忆，而不是向量数据库。这降低了门槛——任何懂 Markdown 的人都能理解和修改 AI 的记忆。

### 4.2 短板与痛点

#### ❌ 1. 单进程架构无法水平扩展
所有 Channel、Agent、Plugin 都在一个 Node.js 进程中。一旦用户量增长（多用户场景），内存和 CPU 都成为瓶颈。不支持多 Gateway 实例协同。

#### ❌ 2. 插件安全模型是"全有或全无"
Native 插件运行在 Gateway 进程中，拥有和核心代码一样的权限。文档明确说"恶意原生插件等于在 OpenClaw 进程中任意代码执行"。这在自托管场景下问题不大，但无法做多租户或不受信任插件托管。

#### ❌ 3. 没有审计日志和合规功能
Delegate 模式虽然有 Tier 1-3 的权限分级，但没有提到审计日志（谁在什么时间让 Agent 做了什么操作）。企业场景下这是必须的。

#### ❌ 4. 移动节点依赖 Tailscale
iOS/Android 节点需要通过 Tailscale 连接。这意味着用户必须：
- 安装 Tailscale
- 管理 Tailscale 账号/魔怔兜底
- 在 Tailscale 私有网络中运行
这增加了一层运维复杂度，尤其在国内网络环境下。

#### ❌ 5. Session 存储在文件系统
JSONL 文件存储在 `~/.openclaw/agents/<id>/sessions/`。这对单用户够用，但如果做多用户服务，会话迁移、备份、并发控制都成问题。

#### ❌ 6. 缺少热门的即时通讯深度集成
虽然支持主流平台，但一些重要功能缺失：
- WhatsApp 没有原生群组管理 API（依赖 Baileys，非官方）
- iMessage 依赖第三方（BlueBubbles）或者 web 界面
- 没有 Signal 支持

#### ❌ 7. 文档以"参考手册"为主，缺少设计决策解释
Architecture 文档描述了"是什么"和"怎么做"，但很少解释"为什么这样设计"。对于想学习系统设计的开发者来说，深度有限。

#### ❌ 8. 插件 SDK 依然复杂
虽然比早期的 hook-only 模式好很多，但创建一个新的 Channel Plugin 仍然需要理解大量 SDK 子路径（100+ 子路径）。这增加了外部贡献者的门槛。

## 五、差异化建议

### 5.1 MVP 核心功能（如果要仿做一个）

如果我们要做一个类似产品，第一版应该专注于：

```
MVP 范围
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ Gateway 进程（Node.js + Hono WebSocket）
✅ 至少 2 个 Channel（Telegram + Discord）
✅ 基础 Agent Runtime（接一个模型：OpenAI 或 Anthropic）
✅ Session 管理（JSONL 文件）
✅ 基础插件注册系统（Provider Plugin）
✅ Config Schema 验证
✅ Control UI（浏览器聊天界面）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**不做的：**
- ❌ 移动节点（第一版不做）
- ❌ 多 Agent（单 Agent 先）
- ❌ 沙箱执行（第一版不做）
- ❌ MCP 集成（后加）

### 5.2 差异化方向

#### 差异化 1：**微服务化架构**（适合团队/企业场景）
OpenClaw 的单进程是它的限制，也是机会。如果我们做成分布式的：
```
┌──────────┐  ┌──────────┐  ┌──────────┐
│Channel   │  │Agent     │  │Session   │
│Service   │  │Service   │  │Service   │
│(横向扩展) │  │(横向扩展) │  │(Redis)    │
└──────────┘  └──────────┘  └──────────┘
```
好处：支持多租户、支持大用户量、支持按服务升级。  
代价：部署复杂度大幅增加。

#### 差异化 2：**真正的插件沙箱**（安全多租户）
用 WebAssembly 或 Docker 容器隔离插件执行，实现：
- 不受信任的第三方插件可以在同一实例运行
- 插件资源使用可计量、可限制
- 插件可以崩溃但不影响核心

#### 差异化 3：**企业级功能**（审计日志 + SSO）
- 完整的操作审计日志（谁在什么时候让 AI 做了什么）
- OIDC/SAML 单点登录
- 基于角色的访问控制（RBAC）
- 合规数据导出（GDPR 等）

#### 差异化 4：**开发者体验优先**
- `npx create-myagent` 一键创建新插件项目
- 热重载插件开发（修改代码无需重启 Gateway）
- Web-based 调试界面（查看 Agent 的思考过程、Token 消耗）
- 插件市场（marketplace）+ 自动发布流程

#### 差异化 5：**更好的中文体验**
- 飞书/钉钉/企业微信一等支持（国内用户）
- 中文文档和社区
- 国内模型一键接入（月之暗面、百度、文心、MiniMax）

### 5.3 技术选型建议

| 组件 | OpenClaw 方案 | 替代方案（适合大团队） |
|------|--------------|---------------------|
| 运行时 | Node.js 单进程 | Go/Rust 微服务 |
| HTTP框架 | Hono（轻量） | Fastify（插件生态更好） |
| 会话存储 | JSONL 文件 | Redis + PostgreSQL |
| 插件隔离 | 进程内（不安全） | WASM + Docker（安全） |
| Agent 核心 | Pi Agent Core | LangChain/Llamaindex（更通用） |
| 向量存储 | 无（纯 Markdown） | Qdrant/Pinecone（可选） |
| 协议类型 | TypeBox | Protobuf（性能更强） |
| 部署 | 本地进程 | Kubernetes（支持多租户） |
